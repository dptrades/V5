# AntiGravity Dashboard — Trading Strategy & Data Quality Audit

Scope: full read of `lib/conviction.ts`, `lib/indicators.ts`, `lib/options.ts`, `lib/news.ts`, `lib/news-service.ts`, `lib/sector-service.ts`, `lib/social.ts`, `lib/tracking.ts`, `app/api/auto-trade/route.ts`, `app/performance/page.tsx`, `package.json`, `scripts/test-earnings-protection.ts`, plus targeted greps against `lib/constants.ts`. All findings below are cited to a specific file and verified against the actual code, not the docs.

## Bottom line

This is a well-built dashboard with some genuinely good ideas (delta-targeted option strikes, earnings-blackout logic, ADX trend-strength filtering, multi-timeframe confluence). But as a *system a trader should risk capital on*, it has three disqualifying problems: the live auto-trade bot trades off the scanner with the weakest guardrails, a meaningful chunk of the "Social Sentiment" signal is either mislabeled or synthetic rather than real social data, and there is no backtest or track record anywhere that validates the scoring weights actually predict returns. None of this is fatal — all three are fixable — but none of it should be trusted with real money until fixed.

## 1. The auto-trade bot uses the wrong scanner

`app/api/auto-trade/route.ts` calls `scanAlphaHunter()`, not `scanConviction()`. That matters because the two scanners are not equally rigorous. `scanConviction` ("Top Picks") applies a quality gate — `technicalScore >= 50 AND analystScore >= 50` (`lib/conviction.ts:499`) — and a sector cap of 6 stocks (`lib/conviction.ts:508`, `lib/constants.ts:106`). `scanAlphaHunter` applies neither, by design (its own code comment says it "intentionally skips this to stay broader") and weights a 30% "Discovery" score that rewards stocks merely for being flagged by a momentum/unusual-activity scanner. The auto-trade route then adds only three filters on top: exclude index/ETF symbols, exclude existing holdings, and require `score >= 50` and `trend === 'BULLISH'`. The net effect: the only system placing real orders is fed by the scanner explicitly built to be less selective, with no sector-concentration control. Four live positions could legally end up in the same sector with no fundamental or analyst-quality floor at all. If the bot is meant to trade, it should be filtering from `scanConviction`'s output (or have its own quality/sector gate inlined), not from the discovery-weighted scanner.

A second, smaller issue in the same file: the `pick.score < 50` filter at `route.ts:144` is a no-op. `scanAlphaHunter()` already returns `sortedFiltered`, which is pre-filtered to `score >= CONVICTION_SCORE_THRESHOLD (50)` (`lib/conviction.ts:890`). The redundant filter isn't wrong, just dead code — worth knowing so nobody mistakes it for a real second line of defense.

## 2. Risk management is fixed-size, not risk-adjusted

Every trade gets exactly $250 (`TRADE_AMOUNT`, `route.ts:28`), regardless of the stock's volatility or how strong its score is — a stock scoring 51 and a stock scoring 95 get identical dollar exposure. Stop-loss and take-profit are flat percentages (-10% / +25%, `route.ts:29-30`) applied uniformly, so a low-volatility mega-cap and a high-beta name get the same stop distance; the options engine, by contrast, already does this correctly by anchoring stops to ATR and EMA50 (`lib/options.ts:464-481`) — that logic just never made it into the equity auto-trader. There's also no portfolio-level circuit breaker: no daily/weekly loss limit, no max-drawdown halt, and no correlation check across the 4 allowed positions. And there's no earnings-date blackout on the equity path — `lib/options.ts` has a real, tested 7-day earnings deferral (`scripts/test-earnings-protection.ts` exercises it and it passes), but that protection only covers option signals. The stock-buying bot can open a fresh equity position the day before an earnings print with no warning.

Recommendation, in order of impact: (1) point auto-trade at `scanConviction` or replicate its gate/cap inline; (2) size positions by inverse volatility or by score magnitude rather than a flat $250; (3) add a portfolio-level sector/correlation cap to the live trader, not just the display scanner; (4) port the earnings-blackout check from `lib/options.ts` into the equity entry path.

## 3. The "Social" score isn't social data — and the Social Pulse widget is synthetic

This is the most important data-quality finding. In `conviction.ts`, the "Social" pillar (20% of Top Picks, 15% of Alpha Hunter — the second- or third-largest weight in the whole model) comes from `calculateSentimentScore(getNewsData(symbol, 'social'))`. Tracing that: `getNewsData()` in `lib/news-service.ts` accepts a `type` parameter (`'news' | 'social' | 'analyst'`) but never branches on it — every call hits the identical Yahoo Finance news search. So the "social" score is literally the same general news-headline sentiment used elsewhere, run through a simple ±10-per-headline counter (`lib/news.ts:52-69`). There is no Reddit, StockTwits, or X/Twitter data anywhere in the scoring pipeline. That's a mislabeling problem, not a fabrication problem — the underlying news + Finnhub-NLP sentiment is real, just not what the label promises.

The separate "Social Pulse" feature (`lib/social.ts`) is a different and more serious problem. Its `sentiment` and `retailBuyRatio` fields are generated by `seededScore()`, a hash function the code's own comment describes as producing "a deterministic pseudo-random float... so the same stock+signal always yields the same value" (`lib/social.ts:39-51`). `computeSentiment()` (`lib/social.ts:58-95`) returns hardcoded baseline constants (0.80, 0.72, 0.68, 0.55, 0.35, 0.48, etc.) selected by keyword matching on a headline, then adds that pseudo-random noise. `topPlatform` is similarly cosmetic — it's set to `'Twitter/X'` whenever the discovery source tag happens to be `'social'`, not because any X/Twitter data was fetched. None of this feeds the auto-trade bot directly (it's a separate discovery widget), but if a user is looking at that page and reading "62% retail buy ratio, top platform Twitter/X" as real telemetry, they're looking at a hash function, not the market. This should either be wired to a real source (Finnhub/StockTwits both have APIs) or relabeled clearly as a heuristic/illustrative estimate.

## 4. Technical scoring has an internal contradiction and some likely-redundant bonuses

`calculateConfluenceScore()` in `lib/indicators.ts` treats RSI<30 as a *bearish* momentum signal ("Price is dumping... NOT a bull signal," `indicators.ts:366-369`) and RSI>80 as *bullish* ("Price is pumping... NOT a bear signal," `indicators.ts:370-374`) — a defensible momentum-following stance. But `conviction.ts` then applies a separate, contradictory adjustment on top: an explicit -10 penalty when `rsi > 80` (treating overbought as a risk to fade). So an overbought stock picks up bullish points in one function and gets some of them clawed back by a different function elsewhere in the pipeline — two parts of the same codebase encode opposite views of the same condition, and they partially cancel rather than reflecting one coherent thesis. Separately, `conviction.ts` stacks a 52-week-high-proximity bonus, an SPY-relative-strength bonus, and (Alpha Hunter only) a volume-surge bonus on top of the base confluence score — three signals that are all just different lenses on "this stock has had strong recent momentum," so a single strong move can earn credit three times rather than the model getting three independent confirmations. None of this means the technical score is useless — the base confluence function (EMA stack, MACD, Bollinger %B, ADX, RSI divergence, candlestick patterns, Fair Value Gaps) is a genuinely above-average feature set for a retail tool — but the redundant stacking and the RSI contradiction both inflate or distort scores in ways that haven't been checked against actual forward returns.

One more soundness note: divergence detection (`indicators.ts:242-304`) is the single highest-weighted signal in the confluence score (+20, more than double any other factor) and is found via a crude 3-bar local-extrema scan over a 21-bar window with no minimum significance filter — it will fire on noise in choppy tape. Given its outsized weight, this is worth either tightening (require a minimum % move between the two compared lows/highs) or down-weighting.

## 5. Fundamental scoring has a units bug that likely neutralizes the leverage penalty

`fundScore -= 10 if debtToEquity > 2.0` appears identically in both scanners (`conviction.ts:317` and `:701`), reading `financialData.debtToEquity` straight from Yahoo. Yahoo's `debtToEquity` field is reported as a percentage (e.g., a company with a 1.5:1 D/E ratio returns `150`, not `1.5`). Compared against a raw threshold of `2.0`, this condition will be true for nearly every company that carries any debt at all — including conservatively levered ones — making the "Safety" check fire almost universally rather than discriminating between safe and risky balance sheets. This should either compare against the actual percentage scale (e.g., `> 150` for D/E > 1.5x) or divide by 100 before comparing. Worth confirming against a live Yahoo payload before changing, but the units mismatch is clear from the API's documented convention.

Separately, every fundamental field defaults to `0` when Yahoo doesn't return it (`|| 0` throughout `conviction.ts`). That conflates "no data" with "data confirms zero," which is a classic quant pitfall: a stock with genuinely missing `revenueGrowth` is scored identically to one with confirmed flat growth, silently denying it the growth bonus it might actually deserve. Given `yahoo-finance2` is an unofficial, undocumented-endpoint wrapper (not a licensed feed, no SLA, prone to missing fields and breakage), this missingness is not rare — it should be tracked separately from "real zero," e.g. by excluding null fields from the score basis rather than zero-filling them.

## 6. Documentation has drifted from the code in ways that matter

`PROJECT_DOCUMENTATION.md` describes a PEG-ratio bonus that doesn't exist anywhere in the code, a PE bonus magnitude that's wrong (+5 documented vs. +10 actual), and an Analyst-score mechanism (counting upgrade/downgrade headlines) that isn't what `conviction.ts` actually does (it uses Yahoo's quantitative `recommendationMean` + price-target upside + EPS-surprise history). The code itself has two stale comments that contradict their own constants: `conviction.ts:503` says "Sector diversity cap: max 3 stocks per sector" while the actual constant is 6 (`constants.ts:106`), and `conviction.ts:512` says "Score >= 75 threshold" while the actual constant is 50 (`constants.ts:103`). None of these are functional bugs — the code runs on the real constants — but they're exactly the kind of drift that causes a future change (by you or an AI assistant) to "fix" the wrong number, because the comment lies about intent. Worth a documentation pass before making further changes to this file.

Also worth flagging precisely so it isn't over- or under-trusted: Alpha Hunter's Yahoo request omits the `earningsTrend` module (`conviction.ts:608`) while its scoring code still tries to read `earningsTrend.trend` for an EPS-surprise bonus — that bonus can never fire for Alpha Hunter. It's dead code, not a live bug, but it means Alpha Hunter's analyst score is quietly less informative than Top Picks' despite looking like the same formula.

## 7. No backtest, no track record, no statistical validation exists

`package.json` has no testing framework (no jest/vitest/mocha), no backtesting library, and no stats/quant package. The only validation artifacts are a handful of `scripts/test-*.ts` files that run manual `console.log`/`console.error` assertions via `ts-node` — useful as smoke tests (the earnings-protection one genuinely exercises real logic and passes), but not a real test suite, and not wired into CI. `lib/tracking.ts` + `app/performance/page.tsx` track individual option recommendations' premium history and flag ±25% moves as PROFIT/LOSS, which is honest bookkeeping — but it's per-trade only. There's no aggregate win rate, no Sharpe/Sortino, no comparison against SPY, and nothing that feeds back into recalibrating the scoring weights. The weight choices themselves carry comments like "Raised from 0.10 — fundamentals now matters more" and "Reduced from 0.40 — discovery still important but balanced" (`conviction.ts:550,553`), which reads as discretionary hand-tuning rather than anything backed by walk-forward testing. For a system whose entire value proposition is "this scoring formula identifies good stocks," there is currently no evidence — inside the codebase — that it actually does, beyond the developer's judgment.

This is the highest-leverage fix available: before changing a single weight, run the existing scanners against, say, 2-3 years of historical point-in-time data and measure forward 30/60/90-day returns of top-decile picks vs. the watchlist average and vs. SPY. Even a rough walk-forward backtest would tell you whether the Discovery-heavy Alpha Hunter weighting actually outperforms the more conservative Top Picks weighting, which right now is an open question the code has no way to answer.

## 8. Data-vendor blending has no cross-validation

Each symbol blends Public.com (live quote), Alpaca primary / Yahoo Finance fallback (OHLCV bars), and Yahoo Finance (`quoteSummary`) for fundamentals — three vendors, three different update cadences, no field-level disagreement detection. If Alpaca's last close and Public.com's live price diverge meaningfully (a split not yet reflected in one feed, a halted/illiquid name, a data outage), nothing flags it; the model just scores on whatever it received. Given that `yahoo-finance2` is an unofficial scraper around undocumented Yahoo endpoints — not a paid, SLA-backed feed — and `lib/sector-service.ts`'s "dynamic" sector map similarly does a raw, unauthenticated `fetch()` against a legacy Yahoo endpoint with generic headers (and silently falls back to the static map on any failure, with no surfaced warning), a lightweight sanity check — e.g., flag/skip a symbol if Alpaca and Public.com prices disagree by more than ~2%, or if a fundamentals field that's normally populated comes back null for a liquid large-cap — would catch a meaningful share of bad inputs before they become a "Top Pick." The static `SECTOR_MAP` itself is also known to drift (GE's data-provider sector classification doesn't match the app's hardcoded "Industrials" tag post-spinoff, as a concrete example), and the watchlists themselves (`MEGA_CAP_WATCHLIST`, `CONVICTION_WATCHLIST`) are hand-maintained arrays with no dynamic membership feed — survivorship and staleness risk that only a human edit can fix.

## 9. The options engine has a fabricated-noise issue and a confidence floor that overstates certainty

`calculateVolatilityProxy()` (`lib/options.ts:44-61`) deliberately adds a pseudo-random offset derived from hashing the ticker symbol, with a comment stating the goal is "so different stocks don't look identical." That noise term can influence the `positionSizingWarning` text ("restrict to 1-2% vs 3-5% of portfolio") whenever real IV/Greeks aren't available from the live chain — meaning a real risk-sizing recommendation can be nudged by what amounts to an arbitrary, name-derived number rather than market data, with no indication to the user that this happened. Separately, the `confidence` score in `generateOptionSignal()` is built from a long chain of additive/subtractive heuristics but is hard-floored at 50 (`Math.max(50, ...)` applied repeatedly) and capped at 99 — so every actionable signal the system ever produces will display "50% or higher confidence" by construction, regardless of how thin the underlying signal actually is. That's not a calibrated probability (the `probabilityITM`/delta field nearby is the closer thing to one); displaying it as "confidence" risks implying more certainty than the math supports. Lastly, `findTopOptions()`'s "Tactical Flow Upgrade" can silently replace a carefully delta-targeted, fundamentals-aware strike with whatever contract has the heaviest raw volume (`options.ts:338-379`) — volume-chasing is a meaningfully different, more speculative strategy than delta-targeting, and the user has no visibility into which one actually produced the displayed recommendation.

## Prioritized fix list

1. Point the live auto-trade bot at the gated, sector-capped scanner (or inline an equivalent gate), and add an earnings-date blackout to the equity entry path — this is the only change that touches real capital with no upside to leaving as-is.
2. Replace flat $250/-10%/+25% sizing with volatility- or score-adjusted sizing, and add a portfolio-level sector/correlation cap to the live trader.
3. Fix or remove the `debtToEquity > 2.0` units bug — verify Yahoo's actual returned scale for a few symbols and correct the threshold.
4. Relabel or properly source the "Social" score and the Social Pulse widget — either wire in a real social-data API (Finnhub/StockTwits both have endpoints) or stop presenting hash-seeded numbers as platform-sourced sentiment.
5. Run a real walk-forward backtest of both scanners against historical data before trusting or further tuning the weights — this is the only way to know if the strategy has edge.
6. Resolve the RSI overbought/oversold contradiction between `indicators.ts` and `conviction.ts`, and consider down-weighting or tightening the divergence detector given its outsized influence.
7. Clean up the stale comments in `conviction.ts` (sector cap, score threshold) and the stale claims in `PROJECT_DOCUMENTATION.md` (PEG, PE bonus, analyst mechanism) so future changes are made against accurate intent.
8. Add cross-vendor sanity checks (price disagreement, missing-field detection) to the data layer, and treat missing fundamental fields as "unknown" rather than zero-filling them.
